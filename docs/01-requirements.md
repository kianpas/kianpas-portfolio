## 1. 프로젝트 목적

여러 곳(이력서, 외부 블로그, GitHub)에 흩어진 **나의 작업물과 글을 한곳에 모아 보여주는** 개인용 사이트다.
별도의 백엔드/DB 없이 **Markdown 파일을 콘텐츠 소스로 삼아 정적으로 렌더링**하는 것을 기본 전략으로 한다.

핵심 가치는 다음과 같다.

- **포트폴리오**: 실무/개인 프로젝트를 정리해 보여준다.
- **블로그**: 학습·경험을 Markdown 글로 기록하고 카테고리/태그로 탐색한다.
- **단순함**: 백엔드·DB·인증 없이, 파일 기반으로 빠르게 운영·배포한다.

본 프로젝트는 **단일 작성자(나)** 전용이다. 회원/인증/댓글 등 멀티유저 기능은 범위에 두지 않는다.

## 2. 기능 범위

### 2.1 구현된 기능

1. **홈** — 소개(Hero) + 최근 글 3개 티저.
2. **블로그 목록** — 페이지네이션. `/blog` → `/blog/page/1` 리라이트.
3. **블로그 상세** — Markdown → HTML 렌더링, 읽기 시간, 이전/다음 글, 카테고리/태그 배지.
4. **카테고리별 / 태그별 글 목록** — `/blog/category/[categoryName]`, `/blog/tag/[tagName]`.
5. **글 검색** — 제목·요약·본문·태그·카테고리 대상 부분 일치(클라이언트 입력 → `/api/search`).
6. **프로젝트 목록** — 실무/개인 구분, "더보기" 페이지네이션.
7. **프로젝트 상세** — Markdown 렌더링, GitHub/데모 링크, 태그.
8. **About** — 프로필/스킬/경력 (`src/data/metadata.ts` 기반).
9. **다크 모드** — `next-themes` + 시스템 설정 연동, FOUC 방지 인라인 스크립트.
10. **디자인 시스템 미리보기** — `/design-system`.

### 2.2 명시적으로 다루지 않는 것

- 백엔드 서버 / 데이터베이스 / ORM
- 사용자 인증·회원가입·댓글
- CMS, 관리자 화면 (글은 Markdown 파일을 직접 추가)
- 실시간 기능, 알림

## 3. 콘텐츠 모델

| 콘텐츠 | 소스 위치 | 식별자(slug) | 상세 |
| --- | --- | --- | --- |
| 글(Post) | `src/posts/*.md` | 파일명 | `docs/03` |
| 프로젝트(Project) | `src/projects/*.md` | 파일명 | `docs/03` |
| 사이트 메타(저자/스킬/경력) | `src/data/metadata.ts` | — | About 페이지에서 사용 |

- 글/프로젝트는 **frontmatter + Markdown 본문**으로 구성한다.
- 파일을 추가/수정하면 곧 콘텐츠가 된다(빌드 시 정적 생성). 규약은 `docs/03` 참고.

---

## 4. 비기능 요구사항 (NFR)

- **정적 우선 렌더링**: 상세 페이지는 `generateStaticParams`로 빌드 타임에 정적 생성한다(`AGENTS.md`의 "Static-first" 제약 유지).
- **콘텐츠 경로 고정**: 콘텐츠 소스 디렉터리(`src/posts`, `src/projects`)와 라우트/slug는 함부로 바꾸지 않는다(`AGENTS.md`).
- **이식성**: Vercel 배포를 기본으로 한다(`docs/04`).
- **유지보수성**: 공통 UI 컴포넌트(`src/components/ui`)와 디자인 토큰을 재사용한다(`DESIGN.md`).
